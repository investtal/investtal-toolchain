# 9cc update — design spec

## Purpose

Add a self-update command to the installed `9cc` CLI so users can pull the latest released version without re-reading the install instructions.

## Context

- `9cc` ships as two thin wrapper scripts: `scripts/9cc.sh` (Bash) and `scripts/9cc.ps1` (PowerShell).
- Installers (`scripts/install.sh`, `scripts/install.ps1`) download the wrapper from GitHub raw URLs and place a symlink/copy on `PATH`.
- A version string is baked into the script (`CC9_VERSION` env fallback). Installers default to `v0.1.0` and point at `raw.githubusercontent.com/investtal/investtal-toolchain/<VERSION>/scripts/9cc.sh|ps1`.
- Git tags already mark releases (`v0.1.0`, `v0.2.0`).

## Chosen approach: re-run installer with latest GitHub tag

Re-use the existing installer. It already knows:
- where `9cc` lives (`CC9_HOME` / `~/.9cc`)
- which bin directory is writable
- how to wire the symlink / copy

The new `9cc update` command only needs to:
1. Detect whether it is running from Bash or PowerShell script.
2. Ask GitHub for the latest tag (`https://api.github.com/repos/investtal/investtal-toolchain/releases/latest`).
3. Re-download and execute the matching installer for the current platform.
4. Report installed version afterward.

### Trade-offs vs other approaches

| Approach | Pros | Cons |
|----------|------|------|
| A. Re-run installer with latest tag | Minimal code, uses existing bin-dir/symlink logic, cross-platform | Two HTTPS round-trips (one for API, one for installer) |
| B. Direct download of `9cc.sh`/`9cc.ps1` from `main` | Fewer round-trips | Bypasses installer, can drift from installer-managed layout; user who installed into custom `CC9_BIN_DIR` may get a broken `PATH` entry |
| C. Add a manifest JSON | Flexible (channels, checksums, rollback) | New artifact to maintain and validate; overkill for current single-script tool |

Option A is the ceiling now. Upgrade to C when 9cc has multiple artifacts, signed releases, or release channels.

## Components

### 1. `9cc update` in `scripts/9cc.sh`

- Parse `update` command in `main()`.
- Probe GitHub API for `tag_name` of the latest release.
- Compare with `$CC9_VERSION`.
  - If same: print `9cc is up to date (<version>)` and exit 0.
  - If different: run the installer for the new tag:
    - `curl -fsSL https://raw.githubusercontent.com/investtal/investtal-toolchain/<TAG>/scripts/install.sh | CC9_VERSION=<TAG> bash`
- After install, exec or print `9cc version` to confirm.

Error handling:
- `curl` failure → print `9cc update: failed to reach GitHub`.
- Missing `jq`/`node`? Prefer pure-shell parsing with `grep`/`sed`/`node` fallback. `node` is already required by `read_setting`, so use `node` if available; otherwise `grep` + `sed` (tag is simple `"tag_name": "vX.Y.Z"`).

### 2. `9cc update` in `scripts/9cc.ps1`

- Parse `update` command in the switch block.
- Query GitHub API (`Invoke-RestMethod` or `System.Net.WebClient`).
- Compare with `$9ccVersion`.
- If newer, run:
  - `irm https://raw.githubusercontent.com/investtal/investtal-toolchain/<TAG>/scripts/install.ps1 | iex` with `$env:CC9_VERSION = <TAG>`.
- Print result.

### 3. Installer behavior on re-install

Both installers already accept `CC9_VERSION` (they override the default with the env var). No installer changes needed unless we want to add a version echo at the end.

### 4. `version` output

Add `9cc version` to help text so users can verify before and after.

## Data flow

```
User runs 9cc update
        |
        v
9cc script detects latest GitHub tag
        |
        +-- same as current version --> "up to date"
        |
        +-- newer --------------------> fetch installer for new tag
                                            |
                                            v
                                    run installer
                                            |
                                            v
                                    print 9cc version
```

## Error scenarios

- No network: report failed API call, leave current install untouched.
- GitHub API rate-limit (no auth): return HTTP 403; report and suggest retry later.
- Installer fails mid-way: installer already has `set -e`/`$ErrorActionPreference = Stop`; 9cc update inherits the exit code.

## Testing

- Update `scripts/9cc.test.sh` and `scripts/9cc.test.ps1` to:
  - Mock `CC9_VERSION` and GitHub API response (e.g., via local file fixture or env override).
  - Assert `9cc update` prints correct up-to-date / updating message.
  - Assert that after update the installed script reports the new version.
- Do not test against live GitHub API in CI.
